## 컨셉: "리릭앱 × SNS"

곡을 재생하면 **미쿠 계정이 가사를 SNS에 실시간 포스팅**하고, 가상의 팔로워들이 좋아요와 댓글을 단다.
**WebGL/캔버스 제로** — 그래픽 기술 없이 순수 DOM UI만으로 대상 수상.

README에 적힌 설계 근거 (그대로 인용할 가치):
> - 세상은 스마트폰 우세이므로 **스마트폰에 최적화**된 리릭앱으로 만들 것
> - **끝까지 곡을 듣게 만들 것**: SNS는 무심코 계속 보게 된다 → SNS×리릭앱이면 끝까지 들어주지 않을까

"심사위원/투표자가 곡을 끝까지 듣게 하는 장치"를 UX 차원에서 설계한 것.

## 아키텍처 (useJotaiPosts.tsx가 심장)

### 가사 → 포스트 변환
- onTimeUpdate 폴링이 아니라 **TextAlive의 `phrase.animate` 훅**에 콜백을 할당 —
  프레이즈 발성 순간에 미쿠 계정이 해당 가사를 포스팅 (공식 권장 애니메이션 API의 정석 사용)
- 새 포스트마다 Chakra toast로 "알림" 표시 — SNS 리얼리티

### 살아있는 타임라인 (핵심 기술이 아니라 핵심 '연기')
- 포스트마다 가상 계정들이 **80% 확률로, 300~5000ms 랜덤 지연 후 좋아요** —
  매 재생마다 타임라인이 미묘하게 다르게 살아 움직임
- **댓글은 프레이즈 startTime을 키로 전부 수작업 집필** (replies.ts 822줄):
  `13726: [{sumire: "わー！"}, {toshizo: "わしの頭もぴかーんしてきた"}, ...]`
- 가사 한 줄 한 줄에 맞춘 리액션 대본 — 콘텐츠 라이팅이 곧 작품

### 페르소나 설계 (10개 가상 계정)
- 연령×성별 매트릭스: 중2 농구소년 ゆうた, "늘 배 아픈 사람", 사우나 프리랜서 "整うマン",
  "하이퍼 키보디스트 토시조"(50대), 일러스트 연습생 すみれ(10대), "초코민트 과격파" 등
- 각 계정에 프로필 문구 + **자기만의 무관한 평소 포스트 히스토리**까지 부여 (data/posts/*.ts)
- 프로필 사진은 본인 제작/허락 이미지 (license 폴더로 출처 관리)

### 사용자 참여
- 사용자도 "ボカロ好き丸" 계정으로 **직접 포스팅·좋아요·댓글** 가능 (usePostWithKey)
- 스와이프 내비게이션, 계정 프로필 화면, 음악 플레이어가 화면 하단에 상주

## 왜 이겼는가

1. **형식의 전복 하나로 승부**: "가사를 어디에 담는가"를 바꿈 — 세상에서 가장 익숙한 UI에
   가사를 흘려보내자, 보는 법을 아무도 배울 필요가 없음. 기술 진입장벽 없이 컨셉만으로 차별화
2. **모바일 퍼스트**: MIKUNAVI 투표자(현장 관객)의 손에 있는 건 스마트폰 — 2023년에 이미 간파
3. **랜덤성으로 생명감**: 좋아요 타이밍·확률의 미세한 무작위가 "녹화 영상이 아니라 지금 일어나는 일"
   이라는 감각을 만듦 — 웹이어야만 하는 이유를 그래픽이 아니라 '시간성'으로 확보
4. **콘텐츠 밀도**: 댓글 822줄 + 페르소나 10명의 세계관 — 코드보다 '대본'에 들인 시간이 보임
5. 깔끔한 상태관리(jotai) — 단, 이건 수상 요인이라기보다 제작 효율 요인

## 2026에 주는 시사점

- **그래픽 물량전이 유일한 길이 아니다**: 2023(DOM UI) vs 2025(WebGL 플라네타륨) 둘 다 대상.
  "익숙한 인터페이스 × 가사"의 데페이즈망은 여전히 유효한 광맥
- 2024 카 내비와 묶으면 **"일상의 익숙한 UI에 가사를 침투시키기"** 계보가 보임
  (SNS → 카 내비). 2026 등가물 후보: 메신저, 날씨 앱, 지도, 음악 스트리밍, 캘린더, 카메라 롤…
- 전곡 대응 관점의 약점도 확인: Miku SNS는 **댓글 대본이 곡에 하드코딩** (1곡 전용).
  전곡 대응하려면 대본형 콘텐츠 대신 데이터 구동형 리액션이 필요 — 우리 제약과의 차이점
